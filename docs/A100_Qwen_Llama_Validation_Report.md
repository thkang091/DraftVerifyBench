# A100 Validation: Qwen and Llama Speculative Decoding Tradeoffs

## Abstract

DraftVerifyBench evaluates when speculative decoding actually improves LLM inference latency and
when it backfires. This report summarizes two reduced CUDA validation runs on the same
NVIDIA A100-SXM4-80GB GPU:

- Qwen2.5-0.5B-Instruct draft model with Qwen2.5-7B-Instruct verifier model
- Llama-3.2-1B draft model with Llama-3.1-8B verifier model

The main result is model-pair dependent. Qwen never reached break-even in this reduced run, while
Llama produced several speedups, including a best observed speedup of `5.3355x`, but still slowed
down in `82.22%` of speculative rows. This supports the central thesis of the project:
speculative decoding is conditional, not universally faster.

## Headline Result

| model pair | rows | speculative rows | mean speedup | median speedup | best speedup | worst speedup | slowdown rate | mean acceptance |
|---|---:|---:|---:|---:|---:|---:|---:|---:|
| Qwen2.5 0.5B -> 7B | 108 | 90 | `0.6017x` | `0.6045x` | `0.8258x` | `0.3827x` | `100.00%` | `0.6679` |
| Llama 3.2 1B -> Llama 3.1 8B | 108 | 90 | `0.9892x` | `0.7931x` | `5.3355x` | `0.4766x` | `82.22%` | `0.6366` |

The important contrast is not just that one run was faster than the other. The important contrast is
that the same benchmark, hardware, generation length, prompt suite, temperatures, and decoding
policies produced different behavior across two model families. That makes the result more useful
than a single negative benchmark.

## Cross-Model Comparison

The same benchmark produced sharply different outcomes.

| prompt type | Qwen mean speedup | Llama mean speedup | Qwen slowdown rate | Llama slowdown rate |
|---|---:|---:|---:|---:|
| code completion | `0.6303x` | `0.7896x` | `100.00%` | `90.00%` |
| open ended | `0.5548x` | `0.7861x` | `100.00%` | `86.67%` |
| structured JSON | `0.6200x` | `1.3919x` | `100.00%` | `70.00%` |

Structured JSON was the clearest Llama win region. Qwen structured JSON still remained below
break-even, even with high mean acceptance. This is the most important project-level finding:
the model pair and workload shape matter, not just the speculative decoding algorithm.

## Why Acceptance Rate Alone Failed

Acceptance rate was not a stable predictor of speedup across model pairs:

| run | acceptance vs speedup corr | draft overhead vs speedup corr | verifier calls/token vs speedup corr |
|---|---:|---:|---:|
| Qwen2.5 0.5B -> 7B | `0.5678` | `-0.5086` | `-0.3896` |
| Llama 1B -> 8B | `-0.3131` | `-0.5429` | `0.0640` |

For Qwen, higher acceptance was associated with better speedup, but no row reached break-even. For
Llama, the relationship inverted in this reduced run because the fastest rows were short structured
JSON cases with relatively low measured acceptance but very low draft overhead. In both model
families, draft overhead had a strong negative relationship with speedup.

The practical lesson is:

> Acceptance rate is necessary but not sufficient. A speculative decoder must save enough verifier
> work to overcome draft-model cost, extra forward passes, and framework execution overhead.

## Correctness And Caveats

Greedy speculative decoding should preserve the verifier-only greedy output. In these runs:

- Qwen temperature-0 speculative rows: `45 / 45` exact matches
- Llama temperature-0 speculative rows: `43 / 45` exact matches

The two Llama mismatches occurred on `code_001` under `static k=4` and `rolling_acceptance`.
Because correctness comes before speed, future Llama claims should either debug those mismatches or
report Llama speedups separately for exact-match rows only. The Qwen run had perfect greedy exact
match under this check.

Other limitations:

- Reduced prompt suite: `9` prompts
- Single repetition
- Single GPU
- Batch size `1`
- Custom Hugging Face eager implementation, not vLLM/SGLang serving
- Greedy speculative validation only; exact stochastic speculative sampling is not implemented
- Non-zero-temperature rows are timing diagnostics, not exact speculative-sampling claims
- No torch profiler trace yet

These limitations are not failures. They define the next experiments needed to separate algorithmic
effects from backend implementation effects.

## Showcase Thesis

The strongest defensible claim is:

> DraftVerifyBench shows that speculative decoding is a conditional inference optimization. On the
> same A100 benchmark, Qwen2.5 0.5B -> 7B never reached break-even, while Llama 1B -> 8B produced
> large speedups on structured JSON but still slowed down in most rows. Acceptance rate alone did
> not explain performance; draft overhead and workload/model-pair behavior were stronger drivers.

That is a stronger systems result than simply reporting a speedup. It identifies when the
optimization helps, when it fails, and what needs to be measured before enabling it.

## Next Work

The next upgrades should be:

1. Add slowdown decomposition and oracle break-even analysis.
2. Add `torch.profiler` traces for baseline, static `k=2`, static `k=4`, and adaptive policies.
3. Compare against Hugging Face assisted generation.
4. Compare against vLLM speculative decoding.
5. Re-run the full static grid after correctness and profiling are in place.

Those additions would move the project from a strong benchmark artifact toward a production-grade
speculative decoding profiler.
